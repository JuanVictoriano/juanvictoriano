Unix/Linux Mastery

Efficiency, system health, and log analysis — the commands you'll reach for every single day.
  • Unix Process Monitoring
    Use htop for a better visual interface if installed — color-coded, scrollable, and lets you kill processes interactively. Fall back to top if htop isn't available.
    top / htop
  • Unix Recursive Log Search
    Quickly finds matching strings across all log files in a directory. Swap "error" for any pattern — combine with -i for case-insensitive matching.
    grep -r "error" /var/log
  • Unix File Permissions
    Sets rwxr-xr-x — owner can read/write/execute; group and others can read and execute. Use 644 for files that shouldn't be executed.
    chmod 755 [file]
  • Unix Disk Usage
    The -h flag outputs human-readable sizes (GB/MB) instead of raw blocks. Add --max-depth=1 to drill into a specific directory's subdirectories.
    df -h
  • Unix Live Log View
    Streams new log entries to your screen in real-time as they're written. Invaluable for watching deployments, monitoring auth attempts, or tailing application errors.
    tail -f [file]
  • Unix Port Audit
    Shows exactly which PID is listening on which port. -t TCP, -u UDP, -l listening only, -p show PID, -n skip DNS resolution for speed.
    netstat -tulpn
  • Unix Repeat Last Command as Root
    Executes the previous command with sudo privileges. The !! expands to the full previous command — a huge time-saver when you forget to prefix with sudo.
    sudo !!
  • Unix Secure Remote Login
    The backbone of remote administration — encrypted shell access to any machine. Add -i ~/.ssh/key.pem for key-based auth and -p [port] for a non-standard SSH port.
    ssh user@host
  • Unix Smart File Sync
    Copies only changed bytes, resumes interrupted transfers, and preserves permissions — far superior to scp for large or repeated transfers. Add --dry-run to preview before committing.
    rsync -avz src/ dest/
  • Unix Service Management (systemd)
    Inspect, start, stop, and restart services. systemctl enable --now [svc] starts a service immediately and sets it to launch on boot in a single command.
    systemctl status [svc]
  • Unix Systemd Log Streaming
    Follow logs for a specific service in real time. Filter with --since "1 hour ago" or -p err for errors only — the modern alternative to grepping through /var/log.
    journalctl -u [svc] -f
  • Unix Locate Files
    Recursively search by name, size, modification time, or permissions. Combine with -exec to act on every match — e.g. -exec rm {} \; to delete results.
    find / -name "*.conf"
  • Unix Change Ownership
    Reassigns file and directory ownership; -R applies recursively. Frequently paired with chmod when fixing permission issues after a deploy or file copy.
    chown -R user:group [dir]
  • Unix Find a Process
    A full snapshot of running processes with CPU and memory usage. Pipe through grep to locate a specific service before inspecting, restarting, or killing it.
    ps aux | grep [name]
  • Unix Warning Terminate Processes
    Try a graceful kill -15 (SIGTERM) first to let a process clean up. Only escalate to kill -9 (SIGKILL) when it's truly unresponsive. pkill [name] kills by name instead of PID.
    kill -9 [PID]
  • Unix Schedule Jobs
    Edit the cron table to automate recurring tasks. Format is min hour day month weekday command. List existing jobs with crontab -l before adding new ones.
    crontab -e
  • Unix Archive & Compress
    Bundle a directory into a gzipped tarball for backups or transfers. Extract with tar -xzvf archive.tar.gz. Flags: c create, z gzip, v verbose, f file.
    tar -czvf out.tar.gz [dir]

PowerShell Proficiency

Automation, Active Directory, and Windows internals — cmdlets you'll use daily in any Windows environment.
  • PowerShell List Stopped Services
    Pipes all services through a filter to surface only those not currently running — faster than scrolling through Services.msc for a quick health check.
    Get-Service | Where-Object
  • PowerShell Remote Port Check
    A smarter alternative to ping — tests if a specific TCP port is open on a remote host. Use this before opening a ticket about connectivity; it tells you exactly where the block is.
    Test-NetConnection
  • PowerShell Filter System Errors
    Cuts through event log noise to show only Error-level entries from the System log. Add -Newest 50 to limit output, or -Source to narrow by application.
    Get-EventLog
  • PowerShell Find Disabled AD Accounts
    Queries Active Directory for all disabled user accounts in one shot. Useful for quarterly access reviews, offboarding audits, or license cleanup before renewals.
    Get-ADUser -Filter
  • PowerShell Force Reboot
    Forces a reboot on a hung or unresponsive machine, bypassing any "are you sure?" prompts. Add -ComputerName [server] to reboot a remote machine.
    Restart-Computer -Force
  • PowerShell Remote Command Execution
    Runs any command or script block on a remote server without opening RDP. Wrap multiple commands in the ScriptBlock — cleaner and faster than logging into each machine.
    Invoke-Command
  • PowerShell The Help System
    Your first stop for any unfamiliar cmdlet — -Examples shows real-world usage. Run Update-Help once to download the latest documentation locally.
    Get-Help [cmdlet] -Examples
  • PowerShell Discover Cmdlets
    Search for cmdlets by name or noun using wildcards. Indispensable when you know what you want to do but can't remember which cmdlet actually does it.
    Get-Command *service*
  • PowerShell List & Search Files
    The ls/dir equivalent. -Recurse walks subdirectories; pipe to Where-Object to filter by size, date, or extension.
    Get-ChildItem -Recurse
  • PowerShell Top Resource Hogs
    Surface the processes consuming the most CPU. Pair with Stop-Process -Name [name] -Force to terminate a runaway task once you've identified it.
    Get-Process | Sort CPU -Desc
  • PowerShell CSV Data Handling
    Read and write CSV files as objects — the backbone of bulk user provisioning, reporting, and AD batch operations. Use -NoTypeInformation to keep exports clean.
    Import-Csv / Export-Csv
  • PowerShell System Inventory
    Query hardware and OS details — disks, memory, BIOS, network adapters. The modern replacement for Get-WmiObject, and it works over WinRM for remote inventory.
    Get-CimInstance
  • PowerShell Warning Enable Script Execution
    Allows locally-created scripts to run while still requiring downloaded ones to be signed. Scope it with -Scope CurrentUser to avoid a machine-wide policy change.
    Set-ExecutionPolicy RemoteSigned
  • PowerShell Secure Credentials
    Prompts for and stores credentials securely for use with -Credential on remote cmdlets. Never hard-code passwords in scripts — this is the right way to handle auth.
    $cred = Get-Credential

Default Network Ports

Memorize these for firewall rules and connectivity troubleshooting — the ports you'll reference on every ticket.
Remote Access & Management
  • Network SSH / SFTP — Secure Shell & File Transfer
    The standard for encrypted remote access and secure file transfer. Should be your default for any remote terminal work — never use Telnet.
    :22
  • Network Warning Telnet — Unencrypted Remote Access
    Transmits everything in plaintext — credentials included. Block it at the firewall. Only relevant for legacy hardware that doesn't support SSH.
    :23
  • Network RDP — Remote Desktop Protocol
    Windows GUI remote access. Always restrict this port to a VPN or specific IP ranges — exposed RDP on the internet is a primary ransomware vector.
    :3389
Web & Directory Services
  • Network HTTP / HTTPS — Web Traffic
    All modern web traffic should be on 443. If you see 80 open in production without a redirect, that's a misconfiguration. Force HTTPS at the load balancer or server level.
    :80 / :443
  • Network DNS — Domain Name System (TCP/UDP)
    Most queries run over UDP; TCP is used for large responses and zone transfers. If DNS is broken, nearly everything breaks — always check this first when connectivity goes down.
    :53
  • Network LDAP / LDAPS — Active Directory
    389 is unencrypted LDAP — prefer LDAPS on 636 in any environment handling credentials. Required for AD authentication, group policy, and directory lookups.
    :389 / :636
  • Network Global Catalog — AD Forest Queries
    Used by domain controllers to serve forest-wide directory lookups. 3268 is plaintext; prefer 3269 (SSL). Open these between sites in a multi-domain forest.
    :3268 / :3269
File & Email
  • Network SMTP — Email Sending
    Port 25 is server-to-server. Use 587 (STARTTLS) for client submission — most ISPs block outbound 25 from residential IPs to prevent spam.
    :25 / :587
  • Network IMAP — Email Retrieval
    143 is standard IMAP (unencrypted); 993 is IMAPS (SSL/TLS). Always configure clients to use 993. IMAP keeps mail on the server — good for multi-device access.
    :143 / :993
  • Network Warning FTP — File Transfer Protocol
    Plaintext file transfer — 21 is control, 20 is data. Credentials travel unencrypted, so prefer SFTP (:22) or FTPS. Avoid on any untrusted network.
    :20 / :21
  • Network SMB — Windows File Sharing
    Used for Windows network shares, printers, and inter-process communication. Block this at your perimeter firewall — it should never be exposed to the internet (EternalBlue).
    :445
  • Network NFS — Network File System
    Linux/Unix network file sharing. Restrict access via /etc/exports and firewall rules — NFS has no built-in encryption, so use it only on trusted internal networks.
    :2049
Databases
  • Network MySQL / MariaDB
    The most common open-source relational database port. Never expose it to the internet — bind to localhost or a private subnet and front it with a firewall rule.
    :3306
  • Network PostgreSQL
    The default for "Postgres." Lock down access in pg_hba.conf and require SSL for any remote connections handling sensitive data.
    :5432
  • Network Microsoft SQL Server
    The default for MSSQL. Named instances may use dynamic ports brokered via the SQL Browser service on UDP 1434 — check both when troubleshooting connectivity.
    :1433
  • Network Oracle Database / MongoDB / Redis
    Oracle TNS listener on 1521, MongoDB on 27017, Redis on 6379. All should be firewalled to app servers only — exposed Redis and Mongo instances are routinely scanned and breached.
    :1521 / :27017 / :6379
Infrastructure & Monitoring
  • Network DHCP — Dynamic IP Assignment
    Server listens on 67, clients on 68 (UDP). If devices can't get an IP, check that DHCP traffic isn't being blocked across VLANs — you may need an IP helper/relay.
    :67 / :68
  • Network NTP — Network Time Protocol (UDP)
    Keeps clocks in sync across the network. Critical for Kerberos auth, TLS validation, and log correlation — clock skew breaks AD logins and SIEM timelines.
    :123
  • Network SNMP — Device Monitoring (UDP)
    161 for polling devices, 162 for traps. Use SNMPv3 with auth & encryption — v1/v2c community strings are sent in clear text and are a common recon target.
    :161 / :162
  • Network Kerberos — AD Authentication
    The authentication protocol behind Active Directory logons (TCP/UDP). If Kerberos breaks, single sign-on fails — nearly always tied to DNS or time-sync (NTP) issues.
    :88
  • Network Syslog — Centralized Logging (UDP)
    The standard for shipping logs from network devices and servers to a central collector or SIEM. Use TCP/TLS (often 6514) in production to avoid dropped messages.
    :514

The SysAdmin Gold Rules

The principles that separate good administrators from great ones — internalize these and they'll save you from disasters.
  • Rule Rule of Least Privilege
    Never give a user — or yourself — more access than is strictly necessary. Use a standard account for daily tasks and elevate only when required. Limits blast radius when something goes wrong.
    01
  • Rule Snapshot / Backup First
    Before hitting Update or Delete, ensure you have a way back. A 5-minute backup saves a 5-hour restore. This applies to config files, databases, VMs, and anything you're about to touch.
    02
  • Rule The Reboot Reality
    If a system is acting "weird," a reboot clears the stack and memory. It's often the most efficient first diagnostic step — eliminates transient issues before you spend hours chasing phantom bugs.
    03
  • Rule Document the "Why"
    Don't just document what you changed — document why you changed it. Future-you (and your teammates) will need the context six months from now when something breaks and no one remembers the original reasoning.
    04
  • Rule Monitor Before It Breaks
    If a system isn't monitored, you'll learn it's down from angry users — the worst possible alert. Set up proactive monitoring and alerting on disk, CPU, memory, and service health before you actually need it.
    05
  • Rule Automate the Repetitive
    Anything you do by hand more than twice is a candidate for a script. Automation removes human error, documents the process as code, and frees you for work that actually requires judgment.
    06
  • Rule Test in Staging First
    Never let production be the first place a change runs. Mirror prod as closely as possible in a staging environment and validate there — your users are not your QA team.
    07
  • Rule Change One Thing at a Time
    When troubleshooting, change a single variable, then re-test. Stacking multiple changes at once makes it impossible to know what actually fixed — or broke — the system.
    08

Salesforce Development & VS Code

Source-driven development with the Salesforce CLI (sf v2), Git version control, and the practices that keep an org deployable — the commands every admin-developer should know cold.
Salesforce CLI (sf v2)
  • SF Authorize an Org
    Opens a browser to log in and stores the connection under an alias. --set-default makes it the target for later commands. For sandboxes, add --instance-url https://test.salesforce.com.
    sf org login web --alias dev --set-default
  • SF List Connected Orgs
    Shows every authorized org, its alias, and connection status. Use sf org display for one org's details (instance URL, access token) or sf config set target-org=dev to switch the default.
    sf org list
  • SF Scaffold a Project
    Creates a new DX project with the standard structure (force-app, config, sfdx-project.json). In VS Code, the same thing runs via SF: Create Project in the Command Palette.
    sf project generate --name MyProject
  • SF Retrieve (Pull) Metadata
    Pulls components from an org into your local project. Use --manifest manifest/package.xml to retrieve exactly what's listed in your package manifest, or --metadata ApexClass for a single type.
    sf project retrieve start --metadata ApexClass
  • SF Warning Deploy (Push) Metadata
    Pushes local changes to an org. Always validate first with --dry-run, and for production add --test-level RunLocalTests. Treat a manual prod deploy as a last resort.
    sf project deploy start --source-dir force-app
  • SF Run Anonymous Apex
    Executes an Apex script from a file (or interactively with no flag — type, then Ctrl+D). Perfect for one-off data fixes and quick logic checks without opening the Developer Console.
    sf apex run --file script.apex
  • SF Run Apex Tests
    Run tests from the terminal with coverage. Add --synchronous for local dev or --class-names MyTest to target specific classes. Salesforce requires 75% org-wide coverage to deploy to production.
    sf apex run test --code-coverage --result-format human
  • SF Run a SOQL Query
    Query records straight from the terminal. Add --result-format csv > out.csv to export results, or --use-tooling-api to query metadata objects like ApexClass and EntityDefinition.
    sf data query --query "SELECT Id, Name FROM Account LIMIT 10"
  • SF Create a Scratch Org
    Spins up a disposable, source-driven org for isolated feature work. Define its shape in project-scratch-def.json. Delete it when done with sf org delete scratch --target-org scratch.
    sf org create scratch --definition-file config/project-scratch-def.json --alias scratch
  • SF Stream Debug Logs
    Watch debug logs live as code executes — invaluable for debugging triggers, flows, and integrations in real time without refreshing the log list. Add --color for readable output.
    sf apex tail log --color
  • SF Open the Org in a Browser
    Launches the default org in your browser, already authenticated — no re-login. Add --path lightning/setup/SetupOneHome/home to jump straight into Setup.
    sf org open
Git Version Control
  • Git Clone a Repo
    Copies a remote repository locally, including its full commit history. The first step when joining an existing Salesforce project or pulling down a teammate's work.
    git clone [url]
  • Git Create a Feature Branch
    Branch off for every piece of work. Keeps main clean and deployable and makes pull requests reviewable. Never commit metadata straight to main.
    git checkout -b feature/my-change
  • Git Review Changes
    status shows what's staged or modified; diff shows the exact line changes. Always review before committing — a metadata retrieve often pulls noisy, unrelated changes you don't want.
    git status / git diff
  • Git Stage & Commit
    Stage specific files (avoid a blanket git add .) and commit with a clear, present-tense message. Small, focused commits are far easier to review — and to revert — later.
    git add [files] && git commit -m "msg"
  • Git Sync with Remote
    Push your branch to the remote and pull teammates' work. Pull (or rebase) frequently — the longer a branch drifts, the uglier the merge conflicts get on shared metadata files.
    git push origin [branch] / git pull
  • Git Shelve Work in Progress
    Temporarily sets aside uncommitted changes so you can switch context — say, an urgent production fix. Restore it later with git stash pop.
    git stash
  • Git Safely Undo a Commit
    Creates a new commit that reverses a previous one without rewriting history — safe on shared branches, unlike git reset. The go-to for backing out a bad change cleanly.
    git revert [commit]
  • Git Inspect History
    A compact, visual view of the commit tree across branches. Narrow it with --author, a date range, or a file path to trace exactly when a change landed.
    git log --oneline --graph
Best Practices
  • Best Practice Version Control Everything
    Treat metadata as source code. Every class, flow, layout, and permission set lives in Git — the org is never the source of truth. This makes changes reviewable, traceable, and reversible.
    01
  • Best Practice Never Deploy Straight to Prod
    Promote changes through a pipeline: scratch/dev → integration → UAT → prod, ideally via CI/CD (GitHub Actions, Gearset, Copado). Manual production deploys are how Fridays get ruined.
    02
  • Best Practice Warning Bulkify Your Apex
    Never put SOQL queries or DML statements inside loops. Write triggers and classes to operate on collections of records so they survive bulk operations and stay clear of governor limits.
    03
  • Best Practice Write Real Test Coverage
    Aim well above the 75% minimum, and write tests that actually assert behavior — not just exercise lines. Use Test.startTest()/stopTest() and test-data factories for realistic, isolated data.
    04
  • Best Practice Use a .forceignore
    Exclude noisy or environment-specific metadata (full *.profile-meta.xml files, org settings) from retrieves and deploys. Keeps diffs clean and avoids clobbering org-specific configuration.
    05
  • Best Practice Mind Your Branching Model
    Adopt a consistent strategy — e.g. feature branches reviewed via pull request into main. Tie branches to work items, require review before merge, and keep main always deployable.
    06
📋 Quick Reference

Unix/Linux commands work on any modern Linux distro and macOS terminal. Some (like netstat) may require net-tools to be installed — on newer systems, ss -tulpn is the preferred alternative.

PowerShell cmdlets require the appropriate modules — Get-ADUser needs the ActiveDirectory RSAT module installed. Run PowerShell as Administrator for any commands that touch services, event logs, or remote machines.

Port numbers listed are defaults — applications can be configured to listen on non-standard ports. Always verify with netstat or ss rather than assuming.

Salesforce CLI commands use the modern sf (v2) syntax; the deprecated sfdx force:* commands have been retired. Verify your version with sf --version and stay current with sf plugins update. Install the Salesforce Extension Pack in VS Code to run most of these straight from the Command Palette.
